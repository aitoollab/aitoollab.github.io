E2a：AI Agent的邮件网关，普通人也能做的AI副业新方向

2026-05-12 · 约2500字 · AiToollab

事情是这样的。

前两天我在 Hacker News 上刷到一个项目，26个星，2026年4月25日才开源，不算起眼。但我点进去看了下 README，然后愣了一下——它在做一件什么事呢？

让 AI Agent 能够接收和处理真实世界的邮件。

26个星，放在 GitHub 的海洋里基本就是一滴水。但我盯着这个项目看了半天，越看越觉得这玩意儿有意思。不是因为它技术多牛，而是因为它解决的是一个真实的问题，而且这个问题背后，藏着一片还没被大多数人看到的机会。

这个项目叫 E2a，Authenticated email gateway for AI agents。

今天就把这个东西是什么，以及它背后普通人能怎么参与，掰开了说说。

先搞清楚E2a是干什么的

说人话，E2a 就是一个专门给 AI Agent 用的邮件网关。

你想想看，现在 AI Agent 多火对吧？Claude Code 能写代码，Cursor 能编程，GPT-4o 能处理文档。但这些东西都有一个共同的问题——它们都是"被动触发"的。你给它们一个任务，它们完成。但如果你想让它们自动去处理一些事情，比如当收到某封邮件时自动回复，或者当收到发票时自动录入系统，它们就抓瞎了。

因为它们没有耳朵，没有眼睛，没有一个"实时接收信号"的入口。

E2a 干的事，就是给 AI Agent 装上这么一只"耳朵"。它主要靠三样东西：

SPF/DKIM 验证。这个东西听起来技术，但实际上就是在帮你验证——这封邮件是不是真的是从那个地址发来的，防止伪造。现在垃圾邮件满天飞，这个验证机制就相当于给邮件上了一道锁。

HMAC 签名。这个是用来确保邮件内容没有被篡改的。你收到一封邮件，怎么知道里面的内容在传输过程中没被人动过手脚？HMAC 签名就是那个"数字指纹"。

Webhook + WebSocket 分发。这个是核心。当 E2a 收到一封邮件，它会实时通过 Webhook 或 WebSocket 把邮件内容推送给你的 AI Agent，让 Agent 立刻知道"来邮件了"，然后立刻去处理。

听起来不复杂对吧？但这个"看起来简单"的东西，其实填补了一个很大的空白。

为什么这个东西值得关注

我跟你说，我第一眼看到 E2a 的时候，脑子里冒出来的不是"这技术多厉害"，而是——

这个东西有需求，而且需求还没有被很好地满足。

你想，现在有多少人在做 AI Agent 开发？太多了。但大多数 AI Agent 做的东西，都是"你问我答"或者"你给我一个任务我完成"。真正能让 Agent 在真实业务场景里跑起来的，比如自动处理客户邮件、自动录入订单、自动触发内部审批流程——这些场景的实现路径，一直没有一个好用的基础设施。

E2a 至少把邮件这个口子给堵上了。

而且这东西是开源的，用 Go 写的，部署起来不算难。GitHub 上有 CLI 工具，有 SDK，支持多语言接入。

然后我去查了一下，发现他们还有商业化的规划——免费版 + 付费版。这意味着做这个的团队，是认真想把这个东西做成生意的。

一个东西有人在认真做，而且方向是对的，需求是真实的——这往往就是普通人最容易入局的时候。

普通人能怎么参与

说到这儿，可能有朋友会想：这是开发者的事吧？我又不会写代码。

不是的。我是真的觉得，这个方向，普通人能参与的方式比你想象的多。

先说技术相关的。

第一个，做 E2a 的场景化集成服务。

E2a 现在是一个"通用邮件网关"，但不同的行业、不同的业务场景，需要的是完全不同的处理逻辑。

比如，一个跨境电商的卖家，每天可能收到几十封客户询盘邮件。这些邮件里，有问物流的、有问退款的、有问产品的。以前只能人工一个个回。现在有了 E2a + AI Agent，可以实现：邮件来了 → Agent 自动识别意图 → 自动回复或者分类处理。

但卖家自己不会搭这套系统啊。他需要有人帮他搭。

类似的场景还有：律所的邮件分类、猎头的候选人沟通、财务的发票处理、外贸公司的询盘管理……这些场景里，E2a 作为一个底层能力，上层需要有人做定制化的集成。

这类服务，现在基本是空白。

第二个，围绕 E2a 写教程和做内容。

这是更普适的一个方向。

E2a 是个新项目，中文互联网关于它的内容几乎为零。你去搜"E2a AI Agent邮件"，出来的要么是英文的 GitHub README，要么什么都没有。

这就是窗口。

你可以写"E2a入门教程：3步让AI Agent自动处理邮件"，可以写"用E2a + Claude做邮件自动回复"，可以录视频，可以写小红书帖子，可以做知识星球。

关键是，现在做，竞争对手少。等大家都反应过来了，红利就没了。

第三个，做垂直场景的 E2a 模板。

E2a 的团队在 GitHub 上放了 CLI 和 SDK，方便开发者做二次开发。但对于非技术人员来说，直接用代码还是有点门槛。

这里有一个机会：把 E2a 跟具体场景结合起来，做成"开箱即用"的模板。

比如，一个专门给"跨境电商卖家"用的 E2a 模板，配置好了邮件分类、自动回复、订单录入的流程，卖家只需要填入自己的 API Key 和邮箱配置，就能直接用。

这类模板，可以收费。

我的判断

说真的，我也不知道 E2a 这个项目最后能不能成。开源项目变数太多，团队能不能持续投入，市场能不能起来，都不好说。

但有几个东西让我觉得这个方向值得认真看：

需求是真实的。AI Agent 要在真实业务场景里跑起来，需要真实的触发入口。邮件是一个，E2a 开了个头。这个逻辑是通顺的。

竞争还没有起来。E2a 目前在 GitHub 上只有几十个星，关注度很低。早起的鸟儿有虫吃，这个时间点进去，成本低，机会大。

商业化路径清晰。E2a 团队明确说了有免费版和付费版计划，这意味着他们是认真想做产品的。认真做产品的人，值得跟。

当然，风险也存在。最主要的风险是——这个项目本身会不会被更大的玩家复制？比如 Gmail 自己出了个官方 API 呢？或者某个大厂做了个封闭的邮件 AI 平台呢？

有可能。但说实话，即使被复制，在复制的过程中也会产生大量的定制化和集成需求。早期参与的人，永远比后来者有优势。

我是这么想的：在一个新方向还处于"大多数人不知道，不看好"的时候进去，恰恰是风险最低的时候。因为所有人都还在观望，你已经开始做了。

不怕慢，就怕站。

最后说两句

其实我写这篇文章，不是为了给 E2a 打广告。我打心底觉得，这个项目能不能成还是两说。

我想让你看到的，是这种"基础设施型工具"背后的思维方式。

每次一个新的技术底座出现的时候，总会有一批人冲进去做应用层的东西。2023年的 GPT API，催生了一大批套壳应用和 Prompt 工具；2024年的 AI Agent 框架，催生了一大批自动化工作流；2025年的 MCP 协议，催生了一大批连接器。

E2a 代表的，是 AI Agent 和真实世界数据源连接的这个方向。这个方向，才刚刚开始。

要不要参与，怎么参与，你自己判断。但我想告诉你的是——

机会这东西，看见了是机会，看不见就是噪音。

先看见。然后动手。
